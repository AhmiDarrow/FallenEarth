## Issues

### Issue 1 — Severity: bug
- File: scripts/SaveManager.gd:36
- Description: save_to_game_file always wraps the passed `game_state` arg under a top-level `"game_state"` key inside the saved dict. Callers pass inconsistent shapes: GameState.save_game passes a full `{ "character": ..., "appearance": ... }` object; CharacterSelection passes bare char_data. Load paths (GameState.load_game, MainMenu._list_saves, SaveManager.list_all_slots) all assume top-level "character", "save_name" etc.
- Suggestion: Unify save shape. Either (a) have callers pass raw character and let SaveManager assemble consistently under documented keys, or (b) make SaveManager a thin writer and move assembly to GameState/SaveManager callers. Update load_game + _list_saves to handle both legacy shapes during transition. Add a versioned save schema.
- Status: open

### Issue 2 — Severity: bug
- File: scripts/GameManager.gd:67 (and CharacterSelection.gd:389)
- Description: `_load_scene(..., forward_hub_data=true)` does `await get_tree().process_frame` then `get_tree().current_scene as HubWorld` + direct method call. Called without `await` from CharacterSelection signal handler (scene about to be freed). go_to_hub path mixes direct packed change + deferred in other paths. Fragile under load timing or multiple rapid transitions.
- Suggestion: Prefer GameState as single source for pending character + use a deferred/ready hook in HubWorld (already partially does gs.get_character_data() in _ready). Or emit a signal after scene change completes and let receivers pull. Test thoroughly with F5.
- Status: open

### Issue 3 — Severity: bug

- Suggestion: Wire Load button to call `_list_saves()`, populate a simple popup/list (reuse or add minimal UI), then call GameState.load_game + GameManager navigation on selection. Align dir const with SaveManager.SAVE_DIR.
- Status: open

### Issue 4 — Severity: suggestion
- File: scripts/WorldGenerator.gd:1 + generate()

- Suggestion: Either integrate the Python hexasphere output (export JSON tiles) or evolve WorldGenerator toward real hex grid (use existing biome_rules.py as guide). Update docs to reflect reality or plan. Store world data and render a preview grid in Hub/World view next.
- Status: open
