## 2. What Was Accomplished

### Issue 1: `get_weapon("Scavenger", 0)` returned `{}` in tests 2, 4
**Root cause:** Autoloads are added to the SceneTree root BEFORE `_initialize` of a `-s` script runs, but their `_ready()` (which calls `_load_weapons()` / `_load_armor()` / `_load_accessories()`) is **deferred to the next idle frame**. So `root.get_node_or_null("EquipmentManager")` returned a valid Node reference, but the data-loaders hadn't run yet, leaving `_weapons_data == {}`.

Tests 1, 3, 6 only call `get_max_hp()` (which doesn't read `_weapons_data`), so they incidentally passed. Tests 2 and 4 call `get_weapon()` and got the empty default.

**Fix:** `tools/smoke_v050.gd` now `await process_frame` at the very start of `_initialize()`. This flushes the deferred `_ready` calls so all autoloads have their data loaded before any test runs.

### Issue 2: `use_item("bandage")` returned "Cannot use items." in test 5
**Root cause:** `_spawn_player()` does NOT call `_advance_to_next_turn()` (only `setup_from_encounter` does). So `active_unit_id` is the empty string after `_spawn_player` returns. The test then did:
```gdscript
var unit: Dictionary = cm._get_unit_ref(cm.active_unit_id)  # active_unit_id is "" → unit = {}
unit["hp"] = 30                                            # writes to local empty dict
cm.active_unit_id = unit.get("id", "")                     # unit has no "id" → ""
```
Result: `active_unit_id` was never set to "player", so `is_player_active()` returned false, and `use_item` bailed with "Cannot use items."

**Fix:** Set `cm.battle_phase = cm.BattlePhase.ACTIVE` and `cm.active_unit_id = "player"` BEFORE looking up the unit. The player id is always `"player"` (set in `_spawn_player`). Same fix applied to test 6 (which incidentally passed before because the bug was just enough to keep `use_item` from succeeding — its assertion only checks `ok == false`).

## 3. Key Decisions & Rationale

- **`await process_frame` over manually calling `_ready`:** Defer-to-frame is the standard Godot pattern. Manually invoking `_ready` would skip the deferred-call queue and miss any other autoload-side init. The one-line await is the minimal correct fix.
- **Hard-code `active_unit_id = "player"`:** The player id is set inside `_spawn_player` and never changes. No need to look it up; the test's intent is "the freshly-spawned player is the active unit".
- **Did NOT add a `setup_from_encounter`-like helper for tests.** Tests 5 and 6 need a minimal player; full encounter setup (enemies, CT init, turn advance) is overkill. Keeping the manual spawn is clearer for the unit test.

## 7. Context That Must Survive

- **Autoload `_ready` is deferred to the next idle frame.** This means any test that calls `root.get_node_or_null("/root/SomeAutoload")` and then immediately reads data will see an empty dict. The fix is `await process_frame` (or `await get_tree().process_frame`) at the start of the test, BEFORE looking up any autoload's data. - **Smoke test execution context:** `extends SceneTree` + `_initialize` is the entry point. `_initialize` runs after autoloads are added to the tree but BEFORE their `_ready` fires. So the very first thing any smoke test should do is `await process_frame` (or two, to be safe).
