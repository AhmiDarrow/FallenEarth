## 3. Key Decisions & Rationale

- **Why `TileMapLayer` (not a custom Sprite2D grid):** Godot 4.3's `TileMapLayer` paints 512×512 (~262k cells) at 60 fps with native Y-sorted `MobLayer` stacking, collision built in for BLOCKED tiles, and zero custom draw code. The old chunked renderer was a workaround for the fact that we never had a real `TileSet`.
- **Why delete the old `selected/`-based TileSetBuilder:** the directory structure (`assets/tilesets/<biome>/{selected,ground,debris,vegetation}/...`) is incompatible with `TileSetAtlasSource` (which wants a single atlas texture per source) and was the root of the render-conflict the user reported.
- **Why BLOCKED tile only gets collision:** debris / vegetation / rift_scar remain walkable. This matches `LocalMapGenerator.get_terrain_movement_cost` which returns `-1` only for `TERRAIN_BLOCKED`.
- **Why drop the procedural `_make_circle_texture` entirely:** the user explicitly asked to remove draw-code remnants "unless it is absolutely required." Markers are now a `ColorRect` + `Label` — both stock Godot nodes, no per-frame pixel writes.
- **Why no `MobVisual` placeholder dot:** the v0.2.0 round 2 mob sprite regen already produced visible silhouettes for all 27 mobs. A procedural draw fallback would just re-introduce the draw system we're deleting.
- **Why `view.call("add_marker", ...)` instead of `view.add_marker(...)` in HubWorld:** GDScript strict-type inference can't statically resolve the return type of a method called on a `Node2D`-typed reference, even when the underlying script is typed. The `call` + `as` cast sidesteps the parse error while keeping the call site in `LocalMapView`.

## 7. Context That Must Survive

- **Tile cell size is 24×24** (`TileSetService.CELL_SIZE`); the local map is 512×512 cells = 12,288×12,288 px. Camera centers on `_local_x * 24 + 12`, `_local_y * 24 + 12`.
- **`MobVisual` parent is `MobLayer` (not `MarkerLayer`)** so the y-sort stacks mobs correctly with each other and the player. `MobLayer.y_sort_enabled = true` in the scene.
- **`TileSet` physics layer 0** is required for the BLOCKED collision polygon to register. The TileSetService calls `ts.add_physics_layer()` **before** `ts.add_source(source)` — this order is mandatory in Godot 4.3, otherwise `add_collision_polygon(0)` throws "physics.size() = 0".
- **`Image.get_format()` for some biomes' PNGs returns non-RGBA8** (the error "Condition 'format != p_src->format' is true" appears in atlas blit). The service now calls `img.convert(Image.FORMAT_RGBA8)` before resize — this is a hard requirement, not a defensive extra.
- **TileSetAtlasSource method call order** (Godot 4.3): create source → `create_tile` for each atlas coord → add source to TileSet → THEN `source.get_tile_data(coords, 0)` works and the returned `TileData` reflects the TileSet's physics layer count.
- **`ResourceLoader.exists("res://assets/tilesets/<biome>/ground.png")` returns false until Godot has imported the folder.** After running `tools/generate_tiles.py`, you must run `godot --headless --path <project> --import` to generate `.png.import` files before any `-s` script can load them. The generator itself does not generate `.import` files.
- **`.gdignore` trick:** the editor's filesystem watcher corrupts partial PNG writes if the editor is open. The Python generator places `assets/tilesets/.gdignore` before writing and removes it after. Don't run the generator with the editor open; or if you must, the .gdignore handles it.
- **`ProjectSettings` autoload `BiomeTilesets` is gone** — if any script in a fresh checkout references `/root/BiomeTilesets` it will be null. (None do, after this refactor — verified.)
- **`@onready` doesn't fire in `SceneTree._initialize` for a freshly added node** — the smoke test uses `await process_frame` after `add_child` to ensure `_ready` has run before calling `configure()`. Real game code (which goes through the main scene tree) does not have this issue.
