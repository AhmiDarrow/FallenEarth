## 3. Key Decisions & Rationale

- **CookingTable is its own layer (`StationLayer`), not part of `SettlementLayer`:** the settlement layer is for faction towns (one per hex, gated by L50 for Riftspire). Cooking tables are local placeable structures, conceptually like HarvestNodes. Separating layers keeps the render order predictable and the hit-test queries clean.
- **`StationLayer` is generic (`station_layer`), not cooking-specific:** future stations (Worktable, ArmorTable, Blacksmith — all deferred from Phase 3) can reuse the same layer. `get_cooking_table_at` filters by `has_method("get_station_id")` so the layer can hold any station type.
- **Mob drops use `qty_min`/`qty_max` rolled in the loot, but the current `LootRoller.roll` only supports fixed `qty`.** For v0.6.0 we use fixed quantities (1, 2, or 3 depending on mob). A `qty_min`/`qty_max` range is a small follow-up if needed.
- **Station UI uses Esc to close (not E).** E is the world-interact key; using it for UI close would cause double-fires. Esc is the universal "back" key in our menu conventions.
- **Recipes live in `data/recipes.json` (not a separate cooking file).** Keeps a single source of truth for all crafting. Filtering by `station` is a one-line query.

## 7. Context That Must Survive

- **The `StationLayer` is generic.** Future station types (Worktable, ArmorTable, Blacksmith) go in the same layer. `get_cooking_table_at` filters by `has_method("get_station_id")`; use `get_station_at(station_id)` for the generic version (follow-up).
- **`map_data["cooking_tables"]` is the data key.** LocalMapGenerator should emit this list to spawn cooking tables on the map. Each entry: `{x: int, y: int, station_id: "cooking_table"}`.
- **Mob drops format: `drops: [{item_id, chance, qty}]`.** `chance` is 0.0-1.0; `qty` is fixed (not a range, for v0.6.0). `LootRoller.roll` reads this and rolls per-mob.
- **The CookingTableUI Esc handler uses `_unhandled_key_input`.** The UI is added to HubWorld, so it processes input before the HubWorld fallback. Make sure the UI's `_unhandled_key_input` calls `get_viewport().set_input_as_handled()` so the keypress doesn't bubble back to HubWorld.
- **The v0.5.0 design for `_spawn_player` (reads from autoload's `pm.level`, not snapshot) still applies.** Any new test that spawns a player at a specific level must set the autoload's level first.
- **`has_method` on a GDScript Script object doesn't reliably expose underscore-prefixed methods.** When testing for the existence of a private helper like `_adjacent_cooking_table`, read the source file directly with `FileAccess`.
