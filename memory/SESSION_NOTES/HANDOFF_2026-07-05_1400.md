## 2. What Was Accomplished

### Per-class weapon stats (the user's key reminder)
**Before v0.6.0:** `em.get_attack` returned `weapon_damage + str_mod` for everyone. A Technician with an int-scaling pistol got `weapon_damage + 0` (the weapon's `{"int": 1}` mod was ignored). Same for Riftbinder (int+wis), Survivor (con), Warden (str+con). **All class weapons effectively used str.**

**v0.6.0 fix:** `em.get_attack` now sums the weapon's `stat_mods` (which is set from the class's `stat_mods` in `data/weapons.json`):
- Scavenger blade → `{"str": 1}` → +1 from weapon
- Technician pistol → `{"int": 1}` → +1 from weapon (NOT str)
- Survivor rifle → `{"con": 1}` → +1 from weapon (NOT str)
- Striker heavy_blade → `{"str": 2}` → +2 from weapon
- Riftbinder focus → `{"int": 2, "wis": 1}` → +3 from weapon
- Warden shield_hammer → `{"str": 1, "con": 1}` → +2 from weapon

Each class benefits from its own primary stats. Weapons with multiple mods (Riftbinder, Warden) get the full benefit. **All other equipment stat mods (armor, accessories like iron_grip str+2) also contribute to attack**, so an iron_grip in acc1 adds +2 to the attack stat.

### Dynamic equipment in damage calc
**Before v0.6.0:** `_spawn_player` set `unit.attack = str/2 + 2 + em.get_attack` (a one-shot snapshot at spawn time). Equipment changes during combat had no effect until the player re-spawned.

**v0.6.0 fix:** `_spawn_player` stores `unit.attack = str/2 + 2` (base only, no equipment). `_effective_attack` and `_effective_armor` now call `em.get_attack(unit_id)` and `em.get_defense(unit_id)` at damage time. **Equipping/switching weapons mid-combat immediately affects the next attack.** Same for armor.

### Three new consumables
- **`mana_potion`** — restores 25 MP on use
- **`cooked_meat`** — heals 15 HP + grants +1 attack buff for 3 turns
- **`antidote`** — heals 10 HP (placeholder for status-cure when those land)

`use_item` was refactored to dispatch via a `_apply_consumable` helper with a `match` on `item_id`. Each item's logic is in its own branch; the consume-from-inventory + `_finish_active_turn` is shared.

`bandage` (heal 30 HP) still works. The bandage tests in `smoke_v050.gd` still pass.

## 3. Key Decisions & Rationale

- **Sum all equipment stat mods vs primary stat only:** Considered using `weapon_kind` → primary stat mapping (blade→str, pistol→int, etc.) but settled on summing all mods from all equipment. Simpler, more general, and matches the data's design (Riftbinder's `{"int": 2, "wis": 1}` should benefit from BOTH stats).
- **Dynamic read vs spawn-time snapshot:** Dynamic read means equip changes take effect immediately. Slight cost: one autoload lookup per damage calculation. Acceptable for the turn-based FFT combat loop.
- **3 new consumables (not 4):** Considered stamina_potion (CT-based) but it would require CT manipulation logic. Skipped for v0.6.0; can add in a follow-up if players want it.
- **Cooked_meat buff duration starts at 3:** `use_item` calls `_finish_active_turn` → `_advance_to_next_turn` → `_tick_buffs(player)`. So the buff immediately decrements to 2 by the time the test reads it. This is correct in-game behavior (the buff starts at the START of the player's next turn, and the player's turn is right after use_item).
- **Antidote heals 10 HP only:** No status effects in v0.6.0 to cure. The heal is a placeholder so the item isn't useless. Status-cure wiring lands when status effects ship.

## 7. Context That Must Survive

- **Each class's weapon scales with its own stat mods** (per `data/weapons.json` classes config). Don't re-add a "primary stat only" mapping; the sum-all approach handles Riftbinder's int+wis and Warden's str+con correctly.
- **`_spawn_player` reads from autoload's `pm.level`**, not the snapshot's level. Tests that want a specific level must set `root.get_node_or_null("ProgressionManager").level = X` BEFORE calling `_spawn_player`. (The smoke_v060 `_spawn_player_at` helper does this.)
- **Weapon id format is `weapon_<class>_t<num>` where `num = tier + 1`.** So tier 0 is `t1` (damage 3), tier 1 is `t2` (damage 5), tier 2 is `t3` (damage 7), etc. Don't confuse tier index with id suffix.
- **Defense still uses `em.get_defense` which only counts `con` mod.** That's intentional for v0.6.0 (defense is more about armor than str/int/wis). If we want defense to scale with other stats later, it's a 1-line change in `get_defense`.
- **`use_item` always calls `_finish_active_turn`** which decrements buff turns by 1. So a "3 turn" buff is at 2 turns by the time the test reads it. This is correct in-game behavior.
- **TileMapLayer / autoload `_ready` deferral still applies.** Every smoke test that depends on autoload data needs `await process_frame` at the start of `_initialize`.

## 10. (Remedy's note)

The user's reminder ("make sure not all class weapons use the same stat") was the key insight. The v0.5.0 implementation lumped all class stats into str, which was a real bug — Technicians with pistols were getting the same damage bonus as a weaponless Survivor. The v0.6.0 fix unifies all stat mods across equipment: weapon + armor + accessories all contribute to attack (and similar for defense with con). Each class's weapon now scales with its own stat distribution.

The dynamic read in `_effective_attack` is also a quality-of-life fix: players can swap weapons mid-fight and the next attack reflects the new gear. Combined with the new consumables (mana_potion for casters, cooked_meat for the +attack buff, antidote as a placeholder), combat now feels more like a real RPG.
