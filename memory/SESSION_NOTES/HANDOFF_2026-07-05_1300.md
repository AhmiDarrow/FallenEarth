## 2. What Was Accomplished

### Fix 1: `MissionManager.gd:214` `GameState.mob_key` parser warning
**Root cause:** `GameState.mob_key(...)` was called as a static method on `GameState` (the autoload global), but `GameState.gd` has no `class_name` declaration. GDScript parser flags `Identifier not found: GameState` for static calls. Autoloads are registered as global identifiers, so the variable lookup works, but static-method resolution requires a real class.

**Suspected in the previous handoff as: "may conflict with the autoload singleton of the same name"** — went with Option B (instance call) instead of adding `class_name`.

**Fix:** Three call sites changed from `GameState.mob_key(...)` to instance call:
- `MissionManager.gd:214` (in `build_mission_encounter`) — added `var gs: Node = get_node_or_null("/root/GameState")` lookup.
- `MissionManager.gd:414` (in `_cleanup_mission_world_state`) — used the already-scoped `gs` variable.
- `HubWorld.gd:1370` (in `_seed_local_mobs`) — used the already-scoped `gs` variable.

`mob_key` is declared `static func` in GameState.gd but is callable via instance (GDScript 4 allows this). No data flow change.

### Fix 2: `smoke_phase5.gd:203` assigning `Callable()` to a signal
**Root cause:** `gs.faction_rep_changed = Callable()` tried to assign to a `signal` declaration. Signals cannot be reassigned; the line was a leftover from a "reset signal handler" attempt that never made sense.

**Fix:** Deleted the line. `faction_rep_changed` is owned by `NPCManager`, not GameState, and is never reset in production.

### Fix 3: `smoke_phase5` `spawn_for_hex` RNG flakiness
**Root cause:** The test relied on `randf() > 0.30` (30% spawn chance) × 10 attempts (~97% success), still flaking ~3% of the time.

**Fix:** Seed the global RNG with `seed(12345)` before the loop and bump max attempts to 100. Now deterministic. Verified: 10/10 runs pass.

**Side effect:** Seeding the RNG exposed a pre-existing test bug in the faction rep gate test (see Fix 3b below). The RNG fix is correct; the faction rep bug was always there but was sometimes masked by the `spawn_for_hex` test never reaching the faction rep test in time-sensitive scenarios.

### Fix 3b (bonus): `smoke_phase5` faction rep gate (and `_faction_rep_for` production bug)
**Root cause (deepest):** The faction rep test set `prog.level = 100` on a local `TestProg7` ProgressionManager, but `can_invite` reads from `/root/ProgressionManager` (the autoload). The autoload stayed at L1, so `player_level (1) < min_player_level (5)` returned false at the level check before even reaching the rep check.

**Also found (production):** `_faction_rep_for` had a pre-existing bug at line 312-313:
```gdscript
if _faction_names.is_empty():
    return 0
```
This returned 0 if `_faction_names` was empty, even if the player had high rep. `_faction_names` is loaded in `_ready` from `gs.get_faction_rep().keys()` (empty at startup), so the function was effectively broken until the player first gained rep. Removed the check — always read the rep dict directly.

**Fix:**
1. **Production** (`scripts/PartyNPCManager.gd`): removed the `_faction_names.is_empty()` early-return in `_faction_rep_for`. Added a comment explaining the function's intent.
2. **Test** (`tools/smoke_phase5.gd`): changed the faction rep test to use the autoload's ProgressionManager (`root.get_node_or_null("ProgressionManager")`) instead of creating a local instance. Also changed the template from `legendary_loner` (requires 50 rep + a quest, both gates fail at the same time) to `faction_officer` (requires 10 rep, no quest), which isolates the rep gate cleanly.
3. **Test** (`tools/smoke_phase5.gd`): `get_invite_requirements_text` was also reading from the wrong ProgressionManager (the previous test set it to 100, so the level check passed). Changed to use the autoload and set level to 1 explicitly.

### Fix 4: `smoke_tile_system` rift_scar normalization ERROR
**Root cause:** The test set `terrain[4] = 4` in a 4×4 map and then queried `get_cell_atlas_coords(Vector2i(4, 0))`. But in a 4×4 map, the cell at index 4 is at (x=0, y=1) — the formula is `y * size + x = 4 → y=1, x=0`. The cell (4, 0) is OUT OF BOUNDS and returns (-1, -1), failing the assertion.

**Fix:** Changed `Vector2i(4, 0)` to `Vector2i(0, 1)`. The normalization logic in `LocalMapView.configure` was already correct (line 80-82: `if t < 0 or t > TileSetSvc.TERRAIN_BLOCKED: t = TileSetSvc.TERRAIN_GROUND`); only the test was looking at the wrong cell.

## 3. Key Decisions & Rationale

- **Instance call over `class_name`:** Adding `class_name GameState` could conflict with the autoload global of the same name (per `HANDOFF_2026-07-05_0300.md`). Instance call is explicit, doesn't introduce a class_name conflict, and is the same pattern used in 50+ other places in the codebase that read autoloads.
- **Seed RNG over higher max attempts:** Bumping attempts to 1000 would still flake on the long tail. Seeding makes the test deterministic AND uses a known RNG state for reproducibility.
- **Fixed the production bug in `_faction_rep_for`:** The `_faction_names.is_empty()` check was clearly wrong (the function's intent per its comment is to return the player's max rep, not gate on a cache variable that may be stale). The fix is a 1-line deletion with no behavior change for the well-populated case.
- **Kept the test changes minimal:** Used `faction_officer` instead of `legendary_loner` because the former isolates the rep gate. The test still verifies the same property (rep gate at 10) and is now correctly written.

## 7. Context That Must Survive

- **Autoloads are global variables, not classes.** Static calls like `GameState.mob_key(...)` fail to parse if the script has no `class_name`. Workaround: instance call via `get_node_or_null("/root/...").method()`.
- **Test-side autoloads are shared global state.** A test that creates a local `ProgressionManager.new()` and sets `level = 100` does NOT affect what the production code reads, because production reads from `/root/ProgressionManager` (the autoload). Tests that want to manipulate autoloads must do so on the autoload instance directly.
- **RNG in tests: seed it.** Smoke tests that depend on `randf()` should `seed(N)` for determinism. Otherwise CI flakes.
- **TileMapLayer cell index is `y * size + x`**, not the other way around. A test that does `terrain[4] = X` in a 4×4 map is setting the cell at (x=0, y=1), not (x=4, y=0).
- **Production bug fix in `_faction_rep_for`** was correct and necessary. The function's intent is "return the player's max rep". The `_faction_names.is_empty()` guard was a stale cache check that returned 0 in the common early-game case. Don't re-add it.
- **The autoload GameState's `_faction_rep` is empty at startup** until the player earns rep. Tests that need non-empty rep must set it explicitly (e.g., `gs._faction_rep = {"Iron Accord": 5}`).
- **The 4 pre-existing issues from `HANDOFF_2026-07-05_0530.md` are all resolved.** Update `docs/NEXT_TASKS.md` to remove the P0-* items.
