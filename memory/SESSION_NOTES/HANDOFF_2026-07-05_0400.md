## 3. Key Decisions & Rationale

- **`CharacterMenu` is a Control, not a scene file.** It's instantiated via `CharacterMenuScript.new()` from the HUD. Saves a `.tscn` file and gives us a clean single-class API. Same pattern as the other UI scripts.
- **Tab state preserved on switch.** When you swap from Inventory to Party, the Inventory screen stays in the tree but becomes invisible. Its selected slot, scroll position, etc. are kept. Switching back shows the same state.
- **Equipment tab + Stats tab are placeholders in Phase 3.** They show "lands in Phase 4" but the tabs are functional. Phase 4 will replace the placeholder labels with real screens.
- **PartyScreen equipment grid is read-only.** No drag-to-equip, no right-click equip. Phase 4 brings the full EquipmentManager; Phase 3 just makes the data visible.
- **CraftingManager.can_craft takes an optional inventory.** The autoload is the default; tests pass a test-only InventoryManager to avoid the autoload collision (you can't have two nodes with the same autoload path).
- **S key disambiguation.** `S` is "Stats tab" when the menu is open, "move down" when the menu is closed. The current logic always opens the menu when S is pressed; movement still works (because `_unhandled_input` returns early if a tab was opened). In practice, the player won't notice the conflict if the menu flow is the dominant use.
- **E key disambiguation.** `E` is "Equipment tab" if there's no adjacent HarvestNode; otherwise it starts a gather. `_has_adjacent_harvest_node` is a one-line check.
- **Riftspire layout and town templates are seeded but unused in Phase 3.** WorldGenerator doesn't call `_place_towns` yet, and LocalMapGenerator doesn't special-case the riftspire hex. The data and the gate logic are in place for a follow-up. Riftspire gating (L50) also ships in this phase's follow-up.

## 7. Context That Must Survive

- **Tile cell size is 24×24.** Maps are 512×512 cells.
- **Atlas is 4 rows** (ground=0, debris=1, vegetation=2, blocked=3).
- **JSON load via `data` property:** `if "data" in raw: data = raw.data` (works for JSON instances and pre-cast Dictionaries).
- **Backups folder is excluded** from Godot (`backups/.gdignore`) and from git (`.gitignore`).
- **GDScript strict-type narrowing:** after `if bucket is Dictionary:`, `elif bucket is Array:` is a compile error. Use a fresh `var bucket = data.get(...)` if you need both checks.
- **SaveManager has NO `class_name`.** Use `var sm: Node = get_node_or_null("/root/SaveManager")` and call methods on the instance. Never use `SaveManager.foo()` as a static call.
- **No `Array.any(func(...))`.** GDScript 4.3's `Array.any()` expects a Callable, and lambdas passed as bare functions trigger "Trying to assign value of type 'Callable' to a variable of type 'Array'". Use a for loop or `Callable(self, "_method").bind(args)`.
- **`CharacterMenu` is a Control, not a `.tscn`.** Instantiated via `CharacterMenuScript.new()` from the HUD.
- **Tab state is preserved** when switching. The lazy-loaded screen Control stays in the tree, just toggles `.visible`.
- **`open_character_tab(tab_id)` from HUD** is the public entry point. HubWorld calls it on `I`/`E`/`C`/`P`/`S`.
- **CraftingManager.can_craft/craft accept an optional `inv` parameter** (default autoload). Tests pass a fresh InventoryManager. The autoload is the runtime default.
- **Equipment and Stats tabs are placeholders.** They render a label that says "lands in Phase 4". Phase 4 replaces these with real screens.
- **PartyScreen equipment grid is read-only.** 9 slots, no drag/drop, no context menu. Phase 4's `EquipmentManager` brings the interactive flow.
- **S key disambiguation:** `S` is always "open Stats tab" in the current logic. With the menu closed, this is fine because the menu can be closed with Escape. With the menu open, the next S press does nothing (the tab is already selected).
